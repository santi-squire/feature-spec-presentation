Mobile Commander

One feature, three platforms,
zero documentation

Web Commander and iOS Commander have the same features with different behavior. We're porting them to Mobile — but nothing is documented.

/feature-spec
/upload-jira

The Challenge

No specs exist for what we're building

  • 1.
    Features were built without written specs
    Web and iOS Commander were developed over years. No acceptance criteria, no PM documentation was ever written down.
  • 2.
    The code is the only source of truth
    Permissions, validations, edge cases, business rules — all live in legacy codebases, not in any document.
  • 3.
    Now we need to port them to Mobile
    Without written specs, every engineer reverse-engineers the same features, and QA has no criteria to test against.

Our approach: extract acceptance criteria and feature specs directly from the existing codebases — automatically.

TASK: Create Appointment

TypeStoryPriorityP1StatusPartial

Description

Allows a barber or manager to create an appointment by selecting a client, services, date, and time slot. Supports existing clients and walk-ins.

Roles & Permissions

UserAccessRestrictions
BrandCannot createView-only
ManagerAny barberOwn location only
BarberOwn calendarCannot create for others

Acceptance Criteria

Form opens with today's date and next available slot pre-selected
User can search for existing client by name, phone, or email
Total price updates when services are added or removed
Unavailable slots are grayed out
Success message appears and calendar updates after saving

Testing Notes

• Test with Brand (no create button), Manager, and Barber
• Test with "Online Booking" enabled and disabled
• Test walk-in flow (no client)
• Test on iOS and Android

Integration Testing

1. Create on mobile → verify in Web Commander
2. Confirm client notification sent
3. Test double-booking conflict

Release Notes

Barbers and managers can create appointments from the mobile app with service selection and real-time availability.
